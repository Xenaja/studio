FuelMonitor – fuel-control platform

From product tasks to engineering challenges

Problem class: integrating independent sources · entity resolution with no shared key · multi-tenant · "dirty" data from real equipment Stack: Python, Django, background services (watchdog, scheduler), gunicorn + nginx, systemd Role: in partnership · Status: in production

The class of problems this project solves

Many business tasks boil down to one hard pattern: there are several independent systems, each knows its own part of the truth, and nowhere are those parts brought together.

Data arrives in different formats, through different channels, with typos and gaps – and a shared identifier to link them by simply doesn't exist.

FuelMonitor is a typical example. Two independent accounting systems: one knows how much of the resource was dispensed, the other – how much actually reached the recipient. As long as they're apart, the discrepancy can't be proven. We brought them into a single view so the discrepancy shows up automatically for every operation.

The technical challenges and how we solved them

  • 01Entity resolution with no shared identifier

    Link records from two systems that have no common key, with "dirty" data – typos, different keyboard layouts, duplicates, an identifier buried inside another field. The solution is a multi-pass algorithm: exact match → fuzzy match (typo-tolerant) → probabilistic recovery from indirect signals. The system doesn't silently "swallow" uncertain matches – disputed cases go to an operator for one-click confirmation.

  • 02Integrations that account for an imperfect real world

    Two different channels: real-time file exchange (several formats, auto-recovery after failures) and an external API with hard rate limits (request chunking, quota monitoring, distinguishing "no data" from "error"). Each automated channel has a manual fallback in case of failure.

  • 03Multi-tenancy with guaranteed isolation

    A single instance serves many clients in a "partner → client" hierarchy. Isolation runs through the whole system at the architecture level – you can't see someone else's data even by editing the address bar. It's verified by dedicated automated tests.

  • 04Data-quality control as part of the product

    The system flags suspicious records by rules itself – the operator works through a list of problems instead of reviewing everything. Anomalies and contradictions between sources are surfaced automatically.

  • 05Performance and reliability at real volumes

    Batch processing sped up imports by tens of times (critical at thousands of operations a day). Heavy recalculations run only on real changes. Key steps are logged: when investigating incidents, it's always clear where and what happened.

The result

  • Disparate sources are brought into a single picture – discrepancies show up automatically for every operation.
  • Matching without a shared key works on real dirty data, with built-in manual control of disputed cases.
  • Performance holds up at real volumes; data isolation is guaranteed at the architecture level.
  • 350+ automated tests (parsers, algorithms, access control, isolation, end-to-end scenarios), logging, ready for operation on a live server.
